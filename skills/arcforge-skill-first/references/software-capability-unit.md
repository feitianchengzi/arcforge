# 软件能力单元

维护目标 skill 前使用本文件。目标是先打开 skill 的实现边界，再决定本轮最小落地形态。

## 核心定义

skill 是 agent 面向一类任务的交互入口和工作流契约。做事时先识别需要哪些 skill；进入某个目标 skill 后，再分析它如何实现。它可以只是轻量文档，也可以发展成小型软件能力包。

`skill-creator` 的 `SKILL.md`、`scripts/`、`references/`、`assets/` 结构是基础形态，不是能力建模的终点。使用 Skill First 时，要继续判断目标 skill 是否需要更明确的交互入口、可复现执行、状态、UI handoff、回传机制、已有实现接入和真实任务验证。

一个完整的软件能力单元可以包含：

- 交互入口：什么时候触发 skill、如何理解用户意图、何时澄清、何时停止。
- 工作流协议：步骤顺序、确认点、失败状态、输出格式和交接边界。
- CLI：可复现的本地执行能力，适合扫描、校验、生成、迁移、同步、批量操作和测试。
- Server：长生命周期能力，适合索引、缓存、任务队列、状态订阅、跨会话数据、UI 数据源和回调接收。
- UI：按需出现的交互面，适合选择、预览、diff、冲突处理、视觉审查、复杂表单和批量确认。
- 状态和数据模型：profile、source record、manifest、schema、audit result、task result 或用户选择。
- 测试和模拟：fixture、golden case、dry run、结构检查、子代理前测和复测。
- 回传机制：UI、CLI 或 server 产生的结构化结果如何回到 agent 对话上下文、本地状态或下一步动作。
- 治理边界：哪些阶段交给 ArcForge 的 scan、audit、merge、profile、apply、drift、publish 或 share。

这些能力不是固定清单。不要为了完整而堆栈，也不要因为当前只有 Markdown 就把用户需求压扁成文档问题。

## 任务到 Skill

创建或更新目标 skill 前，先回答任务层问题：

- 用户真正要完成什么工作，而不是只想创建什么文件。
- 这个任务需要一个 skill，还是需要多个 skill 协作。
- 如果需要多个 skill，哪个是目标 skill，哪些是辅助 skill、验证 skill 或治理 skill。
- 本轮要创建、维护或观察的是哪个目标 skill，其他 skill 只做上下文辅助、交接、调用还是暂不处理。

不要把 CLI、server、UI、MCP、脚本或内部服务当成和 skill 并列的入口。它们是某个 skill 的实现承载、依赖或底层能力。

## 目标 Skill 适配阈值

Skill First 是元 skill。它不能因为一个泛化 skill 能处理部分上下文，就把该泛化 skill 当成目标 skill。

候选目标 skill 足够匹配，通常需要满足：

- 任务领域匹配：description 或正文明确覆盖用户要做的工作。
- 工作流匹配：有输入收集、假设标注、执行步骤、确认点和停止条件。
- 输出匹配：有该领域可复用的输出结构，而不是泛泛的分析总结。
- 不确定性处理匹配：知道何时补日志、补观测、查配置、向用户澄清或记录假设。
- 验证匹配：另一个 agent 能用它重复执行同类任务，并暴露可迭代的问题。

如果候选 skill 只覆盖技术背景、代码修改、需求文档或通用研究，它只能是辅助 skill。没有足够匹配目标 skill 时，先创建目标 skill 工作副本，再决定是否调用泛化 skill 辅助发现项目上下文或实现承载。

## 实现建模问题

确定目标 skill 后，再回答实现层问题：

- 这个工作是一次性、低频、高频、团队复用，还是会长期演进。
- 哪些步骤需要 agent 判断，哪些步骤适合稳定工具执行。
- 哪些动作需要用户选择、视觉判断、diff 复核或批量确认。
- 是否需要保存状态、来源关系、历史结果、缓存、索引或用户偏好。
- 是否需要可复现的 dry run、plan、apply、rollback 或测试 fixture。
- 当前已有实现承载是什么，缺失实现是什么，缺失实现是否阻碍当前目标。
- 如果需要 UI，UI 的结构化结果如何被 agent 继续使用。

## 发现已有实现承载

实现承载不一定是缺失的。很多能力已经存在，只是没有被目标 skill 引用、描述或治理。

已有实现承载可能在：

- 当前项目源码、脚本、测试、配置或文档中。
- 用户的其他本地仓库或 GitHub 仓库中。
- 已有 CLI、daemon、server、Desktop UI、Web UI 或内部平台中。
- MCP server、connector、API、数据库、任务队列或后台服务中。
- 历史项目、设计稿、数据 schema、fixture、模板、命令手册或运维脚本中。
- 已安装但未被当前 skill 引用的 agent skill、plugin 或本地工具中。

先尝试从当前上下文发现：

- 搜索当前项目中的命令、包入口、scripts、bin、server、apps、tools、schema、fixtures 和 docs。
- 检查已加载 skill、项目本地 `.codex/skills`、`.claude/skills`、`.cursor/skills` 或普通 `skills/`。
- 检查用户请求中出现的仓库名、服务名、命令名、产品名或路径线索。
- 如果目标 skill 的实现需要外部承载，但当前上下文没有足够线索，不要直接宣布缺失。

需要向用户澄清的情况：

- 用户描述了成熟工作流，但当前项目找不到对应实现。
- 任务明显需要 CLI、server、UI、MCP、schema、数据源或测试 fixture，但本地没有线索。
- 新建能力成本较高，且很可能已有内部工具或历史项目可以复用。
- agent 只能看到 skill 目录，无法判断其他仓库或服务状态。
- 继续设计会绑定错误接口、错误数据模型或重复造已有系统。

澄清问题保持短而具体：

```text
这个 skill 背后的实现是否已经有现成仓库、CLI、服务、UI、MCP 或脚本？如果有，请给我路径、命令或入口；我会优先把它接入或维护到这个 skill 里，而不是重新设计。
```

发现已有实现后，先判断策略：

- 接入：已有实现承载足够，只需要在 skill 中描述触发条件、调用方式、输入输出和确认边界。
- 维护：已有实现承载存在但不完整，需要在它自己的代码或配置中修复，再更新 skill 入口。
- 包装：已有实现承载可用，但 agent 难以发现，需要补 skill、schema、manifest、reference 或 agent prompt。
- 新建：确认没有合适能力，或复用成本高于新建最小能力。

## 承载选择

只写 `SKILL.md` 和 reference，适用于：

- 低频、低风险、步骤少的工作流。
- 主要价值是意图路由、注意事项、确认边界或文档化经验。
- 执行可以由 agent 直接使用现有文件和 shell 命令完成。

补 CLI，适用于：

- 需要重复执行、参数化、dry run、结构化输出或测试验证。
- 需要跨项目保持一致，例如 scan、audit、generate、sync、diff。
- shell 命令组合开始变长、容易出错或需要稳定 JSON 结果。

补 server，适用于：

- 需要长期状态、后台任务、索引、缓存、订阅或跨会话数据。
- UI 需要稳定数据源，或用户操作需要回调到本地能力层。
- 单次 CLI 调用无法表达持续任务或多步交互状态。

补 UI，适用于：

- 用户需要看、选、改、排序、对比、确认或处理冲突。
- 结果依赖视觉判断，例如设计审查、diff 复核、批量目标选择。
- 对话式逐项询问会低效、易错或让上下文变脏。

补状态或 schema，适用于：

- 需要追踪来源、profile、用户选择、审计结果、任务结果或版本关系。
- 后续步骤必须消费前一步的结构化结果。
- 需要让 CLI、server、UI 和 agent 对同一对象达成一致。

## UI 回传模型

UI 不应该只是“让用户自己点完再回来描述”。在能力包里，UI 应该产出结构化结果。

推荐模型：

```text
agent 触发 skill
-> skill 判断需要 UI
-> CLI/server 准备候选数据
-> UI 承载选择、编辑、预览或确认
-> UI 输出结构化结果
-> 结果写入本地状态、回调 server，或由 agent 读回
-> agent 基于结果继续执行下一步
```

设计 UI handoff 时至少说明：

- UI 打开什么对象或任务。
- 用户需要完成什么选择、编辑或确认。
- UI 返回的数据结构是什么。
- 返回结果保存在哪里，或如何回到 agent 对话上下文。
- agent 收到结果后的下一步是什么。
- 如果用户取消、超时或结果不完整，流程如何停止。

示例：

```text
ArcForge profile 编辑
UI 输入：scan 得到的 skill 列表和当前 profiles
用户动作：勾选 skills，命名 profile，选择目标 agent
UI 输出：{ profileName, selectedSkills, targetAgent, notes }
agent 下一步：调用 CLI 生成 profile plan；写入前向用户确认
```

## 产物建议

本轮可以只落地最小产物，但要把能力边界说清楚。

- 当前能落地：更新 `SKILL.md`、reference、`agents/openai.yaml`、脚本、schema 或测试 fixture。
- 当前不能落地但必要：记录为产品缺口，说明用户价值、触发场景、输入输出和确认边界。
- 不确定是否必要：向用户澄清，不要假装已经知道。

最终输出中说明本轮选择了哪些承载，哪些承载暂不需要，哪些承载是后续增强方向。
