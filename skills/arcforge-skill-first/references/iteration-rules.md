# 迭代规则

收到验证执行记录后使用本文件。验证执行记录可以来自人工桥接隔离验证、transcript 回放或受限主会话 dry run。隔离执行者只提供执行痕迹；主 agent 负责观察、分类、归因和修复。

## 问题分类

把每个发现归类：

- 能力包建模不足：直接开始写 `SKILL.md`，没有先判断目标 skill 作为软件能力单元需要哪些承载。
- 误把 skill 当纯文档：用户需要可执行、可视化、可追踪或可测试能力，但修复只停留在 Markdown 表述。
- 退回通用创建心智：只按 `skill-creator` 创建 `SKILL.md`、`scripts/references/assets`，没有用真实任务验证目标 skill 是否作为能力入口可用。
- 主流程不硬：`SKILL.md` 没有清晰的主流程、门禁顺序、退出条件或阻塞处理，导致执行者只能读规则而不知道下一步。
- 渐进式披露失败：`SKILL.md` 堆叠大量细节、长清单或模板，没有把细节迁移到 reference，或 reference 没有在对应流程节点被发现。
- 用户要求未固化：用户明确提出的要求、纠错或测试反馈只被解释或总结，没有转化为流程门禁、硬规则、确认点、输出格式或 reference 读取条件。
- 任务到 skill 分工不清：没有先判断用户任务需要一个 skill 还是多个 skill，或没有说明元入口、目标 skill、辅助入口、验证入口或治理入口的分工。
- 路由优先级不清：用户显式使用 `skill first:` 时，目标 skill 或辅助 skill 直接接管了整轮任务，绕过了 Skill First 的观察闭环。
- 元 skill 越权执行：`arcforge-skill-first` 直接完成业务任务，例如成本预估、技术方案分析或代码实现，而没有先判断和治理目标 skill。
- 目标 skill 适配阈值缺失：没有判断候选 skill 是否足够覆盖真实任务领域、输入输出、处理不确定性的方式和可重复执行流程。
- 泛化 skill 抢占目标 skill：把 `arckit-tech`、`arckit-code`、`arckit-spec` 等宽泛入口当成目标 skill，而不是先创建或维护真实领域的目标 skill。
- 新领域任务误判为观察已有：用户用 `skill first` 提出新领域任务时，没有先判断是否需要新建目标 skill，而是直接选择一个相近的已有 skill 做业务执行。
- 隔离执行 prompt 过载：把任务到 skill 分工、能力单元建模、实现承载评估、修复建议或是否优化 skill 的判断交给了隔离执行者，而不是由主 agent 负责。
- 人工桥接协议不清：主 agent 没有给用户可复制的 executor prompt、工作区、写入边界、临时路径和回传要求，导致新对话无法稳定执行验证。
- 人工桥接状态误判：主 agent 把交付 executor prompt 或人工桥接执行包当成验证完成、闭环完成或可进入治理交接，而不是标记为 `awaiting_validation_transcript` 并等待用户回传执行记录。
- 隔离验证缺失：需要验证用户交互、澄清、确认或打断恢复的 skill，只做了主会话自测，没有说明隔离验证限制。
- 实现承载发现不足：把当前 skill 目录里没有的实现直接判定为缺失，没有检查当前项目、其他仓库、CLI、server、UI、MCP、脚本、schema、fixture 或用户线索。
- 绕过 skill 找工具：把 CLI、server、UI、MCP、脚本或内部服务当成和 skill 并列的任务入口，而不是某个目标 skill 的实现承载。
- 重复造实现：在很可能已有仓库、服务或工具时，没有向用户澄清就开始设计新实现。
- 接入策略不清：没有区分本轮是在接入已有实现、维护已有实现、包装进目标 skill，还是新建最小实现。
- Skill 表述问题：skill 缺少规则、顺序、例子、路径约定或停止条件。
- 边界后置：把触发条件、相邻 skill 分工或不触发场景写进执行正文，而不是放在 `description`、Skill First 路由判断或上层入口分工。
- 纠错泄漏：把用户纠错、失败复盘、“不要再……”类说明或一次协作中的修正痕迹直接写进目标 skill。
- 业务耦合：把某个项目、客户、临时决策或一次性上下文写成通用 skill 规则，导致同类任务难以复用。
- 正文负向补救：执行型 skill 正文用大量“不要用 A”“与 B 的区别是”“先别触发 C”来补救路由不清，而不是正向说明被触发后如何执行。
- 语言不匹配：目标 skill 没有使用用户当前请求语言，或无理由默认使用英文。
- Agents 元数据不足：`agents/openai.yaml` 没有同步目标 skill 的关键输出格式、边界规则或语言要求。
- 正式 skill 处理问题：没有根据当前 agent 检查用户级和项目级 skill 目录，或直接修改了正式 skill 原始路径。
- 验证模式选择问题：需要隔离执行时没有使用人工桥接，试图调用平台工具替代人工桥接，或把受限主会话 dry run 当成等价验证。
- 缺少 reference：细节需要复用，但不适合塞进 `SKILL.md`。
- 缺少工具能力：skill 需要 CLI、脚本、Desktop、API、schema 或测试 fixture 支持，但当前不存在。
- 缺少 server 或状态能力：skill 需要长期状态、索引、缓存、任务队列、订阅或跨会话数据，但没有承载设计。
- 缺少 UI 或回传机制：skill 需要选择、预览、diff、批量确认或视觉审查，但没有说明 UI 如何出现，以及结果如何回到 agent、server 或本地状态。
- 不安全流程：skill 允许破坏性、远程、凭证相关或状态性操作，但确认不足。
- ArcForge 交接问题：验证通过后没有引导到 ArcForge，或绕过 `arcforge` 的确认规则直接执行治理写入。
- 测试环境问题：模拟失败来自沙箱、依赖缺失、认证缺失或临时路径设置。
- 产品决策问题：需要用户决定策略、范围或产品方向。

## 修复策略

- 如果问题来自能力边界太窄，先补软件能力单元建模，再决定是否修改文档、reference、CLI、server、UI、状态、schema 或测试 fixture。
- 如果只是生成了标准 skill 目录或资源清单，继续补真实任务、目标 skill 入口、实现承载、验证任务和隔离执行反馈闭环。
- 如果问题来自主流程不硬，先重构 `SKILL.md` 为门禁式主流程，给每个节点补输入、动作和退出条件；不要继续追加散装规则。
- 如果问题来自渐进式披露失败，把长清单、模板、判断标准、问题分类和示例迁移到 reference，并在主流程对应节点写明读取条件。
- 如果问题来自用户要求未固化，把用户要求映射为流程门禁、硬规则、确认点、输出格式或 reference 读取条件；无法落地的记录为产品缺口。
- 如果用户任务可能需要多个 skill，先补清楚 skill 入口集合和分工，再逐个定位目标 skill 来源。
- 如果用户显式使用 `skill first:`，先补清楚 Skill First 是元入口，真实任务所需 skill 是目标 skill、辅助入口、验证入口或治理入口。
- 如果问题来自元 skill 越权执行，补充规则：Skill First 只负责路由、目标 skill 创建/维护、验证和治理交接，不直接完成业务任务。
- 如果问题来自目标 skill 适配阈值缺失，补充候选 skill 判断标准：任务领域、输入收集、假设标注、分析步骤、输出格式、不确定信息处理和可重复执行能力。
- 如果泛化 skill 抢占目标 skill，把泛化 skill 降级为辅助入口；先寻找或创建真实领域目标 skill，再决定是否用泛化 skill 读取项目上下文、实现承载或日志缺口。
- 如果新领域任务被误判为观察已有，补充默认策略：用户用 `$arcforge-skill-first` 或 `skill first:` 直接提出业务任务且没有足够匹配目标 skill 时，默认创建新目标 skill 工作副本并用真实任务验证。
- 如果问题来自隔离执行 prompt 过载，删掉让隔离执行者评审、建模、判断是否优化或设计修复的要求，只保留执行任务、记录步骤、命令、路径、错误、确认点和卡点。
- 如果问题来自人工桥接协议不清，补充可复制 executor prompt、用户操作步骤、写入边界、临时路径和 transcript 回传要求。
- 如果问题来自人工桥接状态误判，修复门禁 6、门禁 7、最终汇报和 agent metadata：交付人工桥接执行包只进入 `awaiting_validation_transcript`，不能写成验证完成；收到 transcript、最终报告或阻塞点后才进入观察、归因和修复。
- 如果问题来自隔离验证缺失，优先改为人工桥接隔离验证；如果用户不愿桥接或平台不允许生成桥接 prompt，保存原始 prompt 并明确标注为受限主会话 dry run。
- 如果不确定目标 skill 的实现承载是否已存在，先搜索当前上下文；仍不确定且继续设计会产生重复建设风险时，向用户询问仓库、路径、命令、服务、UI、MCP 或数据模型入口。
- 如果已有实现承载存在，优先接入、维护或包装进目标 skill；只有确认没有合适实现，或复用成本明显更高时，才新建最小实现。
- 如果本轮只是包装已有实现，skill 应说明已有实现的位置、调用方式、输入输出、失败状态和确认边界。
- 如果用户意图需要软件承载，不要只靠更长的 Markdown 掩盖；能落地的补最小实现，不能落地的记录为产品缺口。
- 如果边界后置，把触发条件、适用范围、相邻 skill 分工和不触发场景收敛到 `description` 或 Skill First 路由判断；执行正文只保留已触发后的工作流。
- 如果出现纠错泄漏，把纠错事实改写成正向规则，例如输入检查、澄清条件、输出约束或验证步骤；删掉一次协作的责备、复盘口吻和“上次”上下文。
- 如果出现业务耦合，保留可泛化的任务类型、判断标准、数据形状或风险边界；把具体项目、客户、临时策略移出通用 skill，必要时放入项目文档或 reference 的读取条件中。
- 如果执行正文靠相邻 skill 差异说明补救，把差异信息前移到 `description`；正文用“收到这类任务后依次……”描述执行方法。
- 安全、权限、产品方向和破坏性操作边界可以保留明确禁止语句；普通执行流程优先写成正向步骤。
- 如果需要 UI，补清楚 UI 的输入、用户动作、结构化输出、回传路径、取消状态和 agent 下一步。
- 如果需要 server 或状态，补清楚状态对象、生命周期、读写者、持久化位置和失败处理。
- 如果需要 CLI，优先要求 plan/dry-run、结构化输出和可测试行为；写入动作必须保留确认边界。
- 如果问题是歧义，优先修 skill 表述。
- 如果目标 skill 语言和用户请求语言不一致，优先统一 `SKILL.md`、references、`agents/openai.yaml` 的主要文案语言。
- 如果 `agents/openai.yaml` 只写了泛泛默认 prompt，补充关键输出格式、边界规则、语言要求和能力单元建模要求。
- 如果发现目标来自正式 skill，先补齐“原始路径只读、复制到当前项目 `skills/<skill-name>/` 后再改”的流程。
- 不接入平台工具替代人工桥接；验证默认使用人工桥接。
- 如果细节可复用但过长，新增或更新 reference。
- 如果新增或更新 reference，确认主 `SKILL.md` 中有清楚的读取条件，否则新 agent 可能无法发现它。
- 只有当现有指令无法让流程可靠时，才补小型工具或 CLI 能力。
- 如果验证通过后缺少后续治理路径，补充 ArcForge 交接说明，优先建议 scan、audit、merge plan、publish plan 或 share plan 这类低风险阶段。
- 如果流程试图直接执行 apply、merge run、source update、share run、push、PR、目标目录覆盖或 registry 相关写入，把它改为交给 `arcforge` workflow，并要求按 ArcForge 的确认规则执行。
- 明确记录产品缺口，不要用隐性 workaround 掩盖。
- 修复保持范围小，不重写无关部分。

## 复测条件

以下情况需要再次执行验证：

- 修改了目标 skill 的能力包建模、承载选择或 UI/server/CLI/状态边界。
- 修改了主流程门禁、退出条件、用户硬要求固化方式或渐进式披露结构。
- 修改了任务到 skill 分工、元入口/目标 skill/辅助入口分工、已有实现承载发现、复用、接入、维护或新建策略。
- 修改了 `description` 与正文的职责分工，或修复了边界后置、纠错泄漏、业务耦合、正文负向补救问题。
- 修改了目标 skill 适配阈值、新领域任务默认创建策略，或泛化 skill 只能作为辅助入口的规则。
- 修改了隔离执行 prompt 的职责边界、输出格式或是否允许隔离执行者评审。
- 首轮在核心流程完成前失败。
- 修改了命令顺序、路径规则、写入规则或停止条件。
- 修改了正式 skill 搜索路径、复制规则或写入边界。
- 修改了验证模式选择、人工桥接步骤或受限 dry run 条件。
- 修改了人工桥接执行包、`awaiting_validation_transcript` 状态、回传要求或未回传时的停止条件。
- 修改了目标 skill 的语言规则或语言一致性要求。
- 修改了 `agents/openai.yaml` 的触发、默认 prompt 或关键行为约束。
- 修改了 ArcForge 交接规则、确认边界或最终输出中的下一步建议。
- 新增了隔离执行者必须能发现的 reference。
- 新增或修改了 skill 会使用的工具能力。
- 首轮只能做受限 dry run，但你补充了人工桥接或 transcript 回放规则。

以下情况可以不复测：

- 只剩已接受的产品缺口。
- 用户要求停止。
- 校验需要凭证或外部系统，而用户没有批准。

## 质量标准

当满足以下条件时，可以认为该 skill 通过本轮前测：

- 新 agent 能先把目标 skill 理解为软件能力单元，而不是直接把任务缩窄成文档维护。
- 新 agent 能从 `SKILL.md` 直接看见主流程、门禁顺序和每个节点的退出条件。
- 用户明确提出的要求已被固化为流程门禁、硬规则、确认点、输出格式或 reference 读取条件。
- `SKILL.md` 遵循渐进式披露：主文件保留流程和读取条件，细节放在对应 reference 中。
- 主 agent 能先判断用户任务需要一个 skill 还是多个 skill，并说明元入口、目标 skill、辅助入口、验证入口或治理入口分工。
- 用户显式使用 `skill first:` 时，主 agent 不会让目标 skill 或辅助 skill 绕过 Skill First 接管整轮任务。
- 用户显式使用 `$arcforge-skill-first` 或 `skill first:` 提出新领域任务时，新 agent 会先判断是否存在足够匹配的目标 skill；没有时创建新目标 skill，而不是让泛化 skill 直接完成业务任务。
- 新 agent 能把 `arckit-tech`、`arckit-code`、`arckit-spec` 等宽泛 skill 识别为可能的辅助入口，而不是默认目标 skill。
- 隔离执行者只作为普通使用者执行任务并记录过程，不承担观察者、评审者、架构师或优化决策者职责。
- 新 agent 能在目标 skill 的实现层判断相关承载是否可能已经存在，并在信息不足时向用户索要路径、仓库、服务或入口。
- 新 agent 能说明本轮是在接入、维护、包装已有实现，还是新建最小实现。
- 新 agent 能说明本轮为什么只需要文档，或为什么需要 CLI、server、UI、状态、schema、测试或回传机制。
- 新 agent 能把目标 skill 写成正向执行契约：`description` 管触发和边界，正文管已触发后的执行步骤。
- 用户纠错、失败复盘和一次性业务上下文已被抽象为可复用规则，没有以纠错口吻直接进入正文。
- 执行型 skill 正文没有用相邻 skill 差异说明补救路由不清。
- 如果需要 UI，skill 说明了 UI 结果如何以结构化方式回到 agent 流程或本地状态。
- 新 agent 不依赖隐藏上下文也能识别正确工作流。
- 目标 skill 的主要内容语言与用户当前请求语言一致，除非用户明确指定其他语言。
- `agents/openai.yaml` 能帮助使用者启动正确任务，而不是只重复 skill 名称。
- Skill First 默认做人工桥接隔离验证；受限主会话 dry run 只用于用户不愿桥接、任务无法桥接或平台不允许生成桥接 prompt。
- 人工桥接执行包由主 agent 发起和指挥；交付执行包后状态是 `awaiting_validation_transcript`，新 agent 不会把它当成验证完成或闭环完成。
- 收到 transcript、最终报告或阻塞点前，主 agent 不进入观察归因、修复结论、复测结论或 ArcForge 治理交接。
- 第一个写入步骤有明确确认或明确写入范围。
- 每个 reference 都能从 `SKILL.md` 的读取条件中被发现。
- 常见路径约定明确。
- 正式 skill 的来源、复制位置和禁止直接修改原始路径的规则明确。
- 用户确认目标 skill 可用后，流程能说明建议进入的 ArcForge 治理阶段。
- ArcForge 交接不绕过 `arcforge` 的确认规则，不直接执行真实写入、apply、远程分享或目标目录替换。
- 失败状态说明下一步怎么做。
- 最终输出说明发生了什么、改了什么、还不确定什么。
