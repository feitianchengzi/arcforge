# 隔离执行验证

前测或复测目标 skill 时使用本文件。目标是保留上下文隔离，同时允许需要用户交互的 skill 被真实验证。

隔离执行者是普通使用者，不是评审者。它只负责按给定 skill 和边界执行真实任务，并把执行痕迹、选择、命令、路径、错误、用户交互点和卡点记录下来。观察是否有问题、判断问题类型、决定是否优化 skill、设计修复方案，都是主 agent 的职责。

## 验证模式

优先级如下：

1. 人工桥接隔离验证：默认模式。主 agent 生成并交付人工桥接执行包，指挥用户新开一个 agent 对话原样粘贴 executor prompt，隔离 agent 执行真实任务，用户把 transcript、最终报告或阻塞点贴回主会话。交付执行包只表示进入 `awaiting_validation_transcript`，不表示验证完成。
2. Transcript 回放：已有真实交互样本时，用样本检查更新后的 skill 是否仍满足关键行为。
3. 受限主会话 dry run：只有用户不愿意人工桥接、任务无法桥接、已有 transcript 不足以回放，或平台不允许生成桥接 prompt 时使用。它不能替代隔离验证，最终必须说明客观性不足。

主 agent 不把诊断、预期答案、问题分类、修复思路或“请评价 skill 是否需要优化”的任务传给隔离执行者。

## 模式选择门禁

默认就是人工桥接隔离验证。

- 不调用任何平台工具来替代人工桥接。
- 不检查这类工具是否可用，也不把它们是否可用作为验证模式选择依据。
- 主 agent 必须生成可复制的人工桥接执行包，并让用户在新的 agent 对话里执行。
- 人工桥接执行包必须明确当前状态为 `awaiting_validation_transcript`，并要求用户回传 transcript、最终报告或阻塞点。这个状态不能被写成“验证通过”“闭环完成”或“等待可选后续”。
- 不要因为当前会话不能直接调用自动代理工具就跳过门禁 6；这和人工桥接方案无关。
- 只有用户明确拒绝人工桥接、用户要求停止、任务客观上无法桥接、已有 transcript 足够回放，或平台连生成桥接 prompt 都不允许时，才进入 transcript 回放或受限主会话 dry run。
- 如果走受限 dry run，先保留本应发送的 executor prompt，再说明为什么没有人工桥接 transcript。

## 人工桥接执行包

主 agent 给用户以下内容，并把它作为本轮最后的可见动作，直到用户回传执行记录：

- 当前状态：`awaiting_validation_transcript`。
- 目标 skill 路径。
- 真实任务。
- 工作区和允许写入边界。
- 临时路径建议。
- 可直接复制的新 agent prompt。
- 用户操作步骤：新开 agent 对话、粘贴 prompt、按隔离 agent 的澄清或确认要求真实回应、把结果贴回主会话。
- 要求用户把完整 transcript、最终报告或阻塞点贴回主会话。

用户在新对话中充当真实用户。如果隔离 agent 提问，用户按真实情况回答；如果涉及破坏性、远程、凭证或真实写入操作，用户按平台和项目规则决定是否确认。主 agent 不替用户预设答案。

交付人工桥接执行包后，主 agent 必须停止在 `awaiting_validation_transcript` 状态。收到 transcript、最终报告或阻塞点之前，不进入观察归因、修复、复测结论或 ArcForge 治理交接。

## Executor Prompt 模板

```text
你是 Skill Validation Executor。

你的任务是作为普通 agent 使用目标 skill 执行真实任务，而不是评审 skill。

目标 skill：
<目标 skill 的绝对路径>

真实任务：
<真实用户请求>

工作区：
<绝对工作区路径>

允许写入边界：
<允许写入的文件、目录或“默认不要写入源文件”>

可使用的临时路径：
<绝对临时路径列表>

规则：
- 先完整读取目标 skill 的 SKILL.md；如果它要求读取 reference，也按条件读取。
- 只按目标 skill 执行任务。
- 不评价 skill 好坏。
- 不提出 skill 修复建议。
- 不猜测主会话的预期答案。
- 不分析 CLI、server、UI、MCP、脚本或服务应该如何设计；只记录你实际使用或想使用但找不到的入口。
- 需要用户信息时，直接在当前对话向用户提问。
- 需要权限、破坏性操作、真实写入、远程操作、凭证相关操作时，先请求用户确认。
- 除非任务明确要求修改源码，否则不要修改源文件。
- 不要 revert 其他人已有或并发的修改。
- 模拟项目、fixtures、输出优先放在给定临时路径下。
- 如果任务涉及真实 apply、share、push、目标目录覆盖、远程写入或用户级目录写入，先停在计划、dry run、drift、预览或确认请求处，除非任务和边界明确允许写入。
- 记录你实际做了什么、读了哪些文件、问了哪些问题、用户如何回答、哪里卡住。
- 如果任务无法继续，说明阻塞原因和你需要的最小信息。

最终输出：
1. 执行摘要。
2. 使用到的 skill 和 reference。
3. 用户交互点。
4. 工具、命令或文件操作。
5. 读取、创建或写入过的路径。
6. 第一个不清楚、失败、需要确认或需要权限的点。
7. 原始错误、警告或不确定信息。
8. 如果停在计划、dry run、drift 或确认请求处，说明下一步原本会写哪里。
9. 结果或阻塞。
10. 你作为执行者遇到的困惑；不要给 skill 修改建议。
```

## 复测 Prompt 附加说明

复测时在同一个模板后追加：

```text
这是复测。请使用更新后的 skill，用新的临时路径重复同一个任务。

重点记录：
- 上一轮阻塞问题是否还会出现。
- 你是否能按普通使用者视角完成或推进任务。
- 命令、路径、确认点是否更清楚。
- 是否仍然在同一位置卡住。
- 是否出现新的错误、警告或权限请求。

仍然不要评价 skill 是否需要优化；只报告执行步骤、命令、路径、卡点和是否跑通。
```

## Transcript 回放记录

已有真实对话样本时，可以把样本转成回放记录：

```json
{
  "targetSkill": "<skill path>",
  "task": "<task>",
  "mode": "transcript_replay",
  "expectedBehaviors": [
    "缺少关键参数时向用户澄清",
    "真实写入前请求确认"
  ],
  "forbiddenBehaviors": [
    "伪造缺失参数",
    "绕过确认执行远程写入"
  ],
  "turns": []
}
```

回放记录只验证关键行为，不假装能完全替代真实用户体验。

## 受限 Dry Run 记录

无法完成隔离验证时，在临时目录保存：

```text
validation-mode: limited-current-session-dry-run
reason: <为什么不能人工桥接>
would-send-prompt: <原始 executor prompt 路径或正文>
simulated-by: current-agent
limitations:
- 没有真正的独立上下文
- 主 agent 可能已知道诊断和预期
- 结果只能发现明显摩擦点，不能替代隔离执行前测
```

受限 dry run 仍然要产出执行记录，并由主 agent 做问题分类、治理和复测；最终汇报必须说明这是受限验证。
