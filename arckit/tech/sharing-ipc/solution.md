# 共享与 IPC 方案

## 方案概述

共享与 IPC 方案覆盖 Electron 安全桥接、系统环境检测、GitHub 来源下载、发布计划生成、GitHub 权限预检、共享目标漂移检查、共享执行、技能文件编辑窗口和漂移差异窗口。

渲染层只通过 preload 暴露的方法调用主进程。主进程负责目录选择器、窗口创建、应用数据目录注入和 IPC 参数映射。工作区扫描、配置组应用、来源下载、归并流程和共享推送由 core 与命令编排层完成。

## IPC 边界

preload 暴露 `window.arcforge` 对象。该对象包含选择工作区、扫描工作区、保存配置、查询默认目标、查询环境、下载来源、生成发布计划、生成共享计划、共享项目、共享目标漂移检查、应用配置组、漂移检查、技能文件读写、打开技能文件编辑窗口和打开差异窗口等方法。

主进程通过 `ipcMain.handle` 注册对应 channel。每个 handler 返回 Promise 结果或抛出错误。

IPC handler 不保留与 CLI 不一致的业务分支。需要访问桌面应用数据目录的共享能力通过参数传入桌面应用数据根目录，再调用命令编排层或 core 模块。

渲染层不访问 Node.js API。BrowserWindow 启用上下文隔离并关闭 Node 集成。

## 技能文件编辑 IPC

技能文件编辑 IPC 包含工作区目录树读取、技能目录树读取、文件读取、文件写入和独立编辑窗口创建。

主进程只允许访问当前工作区内的路径。路径解析后位于工作区之外时，系统拒绝请求。

文件读取只支持普通文本文件。超出编辑大小限制的文件和二进制文件不进入编辑器。

文件写入使用 UTF-8 文本写回当前文件。写入完成后，主进程重新读取文件并返回最新内容和修改时间。

独立编辑窗口由主进程创建，启用上下文隔离并关闭 Node 集成。窗口通过 preload 使用同一组技能文件 IPC。

独立窗口接收渲染层传入的编辑上下文，包括来源目录、配置组、技能列表、共享资产列表、当前文件、目录收起状态、滚动位置和国际化文案。该上下文用于保持独立窗口与内置技能页的文件树、过滤器和当前文件一致。

## 默认目标

默认目标由主进程根据当前用户主目录生成。

Codex 目标路径为用户主目录下 `.codex/skills`。

Claude 目标路径为用户主目录下 `.claude/skills`。

Cursor 目标路径为用户主目录下 `.cursor/skills`。

## 环境检测

环境检测执行 `git --version`，并检测 `arcforge` CLI shim、`skillshare`、`npx` 和 `clawhub` 的可用性。

Git 可用时，结果包含平台、架构、Git 可用状态和 Git 版本。

Git 不可用时，结果包含平台、架构、Git 不可用状态和错误信息。

CLI shim 检测结果包含 shim 路径、shim 目录、shim 是否存在、shim 目录是否在 `PATH` 中和可读提示。

桌面端提供 CLI 修复 IPC。修复操作写入或覆盖用户级 shim，并在完成后由渲染层重新请求环境检测。

模型为 `EnvironmentStatus`。

## 来源解析

来源解析接受 SSH Git URL、完整 HTTP Git URL、`github.com/owner/repo`、`owner/repo` 和 GitHub tree/blob URL。

GitHub tree/blob URL 解析为克隆地址、引用和子目录。

非 GitHub HTTP URL 按普通 Git URL 处理，不解析子目录。

来源为空或格式不符合规则时，系统抛出远端地址错误。

共享直接写入时，如果解析出的子目录已经以当前 `sourceDir` 结尾，系统使用该子目录的父目录作为项目根，避免把 `skills` 写成 `skills/skills`。

当 GitHub tree/blob URL 的分支名包含 `/` 时，系统在克隆或拉取远端引用后，用远端分支列表匹配最长可用分支名，并把剩余路径作为子目录。

## 来源下载

来源下载把解析后的仓库克隆到应用数据目录下的 `sources` 缓存。桌面端不使用 `userData/cache` 作为业务缓存根，避免在大小写不敏感文件系统上与 Electron 的 `Cache` 浏览器缓存目录冲突。

缓存目录名称由仓库名称和来源摘要组成。

缓存目录已存在时，系统只获取远端 refs，不自动合并工作区文件。

来源包含引用时，系统检出该引用。

来源包含子目录时，返回该子目录作为项目根路径。若该子目录名为 `skills`，系统返回其父目录作为项目根路径，避免下载后扫描 `skills/skills`。子目录不存在时，系统抛出错误。

远程 Skill 项目使用来源下载结果作为本地根目录。Git 状态与 fast-forward 更新由 `core/source-update` 作为独立 helper 执行。

## 发布计划

发布计划由 core 的 `createPublishPlan` 生成。

计划输入为工作区根目录和可见性。主进程先扫描工作区，再以配置和技能列表生成 `PublishPlan`。

计划包含文件清单、安装命令和检查项，不执行远端写入。

## 共享执行

共享执行先解析远端来源，再扫描当前工作区，并根据用户选择的配置组选出本次共享的技能。

桌面端共享执行通过命令编排层调用 core 共享模块。CLI 的 `share plan`、`share run` 命令和桌面端共享按钮使用相同的参数结构、配置合并规则、Git 执行规则和 `ShareResult`。

共享计划执行 GitHub 权限预检。预检读取 GitHub CLI 可用性、登录状态、仓库权限、默认分支和可用交付方式。计划不写远端。

共享执行使用隔离的共享工作树，不直接在用户当前工作区 checkout 中切换分支或提交。桌面端缓存根目录为应用数据目录，CLI 缓存根目录为用户主目录下 `.arcforge/cache` 或命令传入的 `--cache-dir`。工作树目录由仓库名和仓库地址摘要组成。

工作树不存在时，系统克隆远端仓库。工作树存在时，系统拉取远端引用信息。

基础分支优先使用来源引用。没有来源引用时，系统读取远端默认分支；无法读取时使用 `main`。

基础分支存在于远端时，系统基于远端分支重建本地分支并快进拉取。基础分支不存在时，系统创建本地分支。共享分支默认命名为 `arcforge/share/<project>`，并从基础分支或当前 HEAD 创建。

同一远端仓库和引用的共享操作使用本地锁串行化，避免多个共享任务同时修改同一个本地 checkout。

目标子目录由来源子目录、共享目标模式和项目名计算。直接模式使用来源子目录；按项目名模式追加项目名。

系统清理相对路径中的空段、当前目录段和上级目录段。

## 共享文件写入

系统保存本地共享设置，包括远端仓库、共享目标模式和项目名。

系统写入共享仓库使用的配置。共享仓库配置中的 `teamRepo` 指向可安装的远端项目引用。

非同源共享时，系统确保目标路径下存在技能来源目录，并按本次选择配置组中的技能目录和当前工作区共享资产目录逐项替换。目标技能来源目录中不属于本次共享清单的其它目录不会被删除。

每个技能或共享资产使用相对技能来源目录的位置写入目标，写入前校验目标路径仍位于目标技能来源目录内。

共享资产写入后带有 `.arcforge-owner.json` 归属元数据。目标侧同名共享资产已存在且归属不是当前项目时，系统阻止共享。目标侧旧共享资产没有归属元数据时，系统允许当前项目首次接管并写入归属元数据。

每个技能或共享资产先复制到临时目录，复制成功后再切换到目标目录。复制失败时，目标侧原目录保留。

共享目标同步技能、共享资产和 README。来源 checkout 不写入 `arcforge.config.json`。

目标路径没有 README 且当前工作区存在 README 时，系统复制 README；目标路径已有 README 时不覆盖整份 README。

系统在 README 中插入或替换当前项目专属的 ArcForge 共享区块。该区块包含可见性、桌面端使用流程、配置组和 CLI 安装命令。

## 共享目标漂移

共享目标漂移检查不执行远端写入。普通共享目标会准备或更新隔离共享 checkout，然后根据远端来源、目标模式、项目名和 `sourceDir` 定位共享目标路径。

系统按当前共享配置组选出技能，并把这些技能和当前工作区共享资产与目标路径中的来源目录逐项比较。比较结果复用 `DriftReport`，用于共享页摘要和差异窗口。

同仓库共享目标不做目录对比，而是对当前 Skill 项目在本地 Git 仓库中的相对路径执行 `git status --porcelain`，把待提交变更转换为共享漂移报告。

共享目标漂移可以更新本地缓存 checkout，但不会提交、推送或创建 Pull Request。

## Git 执行

系统对目标路径执行 `git add`。

Git 状态存在变更时，系统使用用户提交信息创建提交。提交信息为空时使用默认提交信息。

Git 状态没有变更时，系统不创建提交，并记录没有文件变更。

本地分支交付不推送远端。直接推送交付把共享分支推送到 origin。目标仓库 Pull Request 交付把共享分支推送到 origin 后调用 GitHub CLI 创建 Pull Request。fork Pull Request 交付创建或复用 fork，把共享分支推送到 fork remote，再向目标仓库创建 Pull Request。

Git 状态没有变更时，系统不推送远端，也不创建 Pull Request。

共享计划结果模型为 `SharePlanResult`，共享执行结果模型为 `ShareResult`。

## 漂移差异窗口

主进程根据 `DriftReport` 渲染只读 HTML 差异窗口。

差异窗口展示配置组、目标目录、总数、变更数、一致数和文件级哈希摘要。

窗口禁用 Node 集成并启用上下文隔离。
