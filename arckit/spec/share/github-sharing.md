# GitHub 共享

## 功能概述

GitHub 共享功能围绕 GitHub-first 工作流生成发布计划，并把 Skill 项目同步到指定 Git 仓库。该功能使个人或小团队可以用 Git 仓库作为技能共享来源，而不依赖托管注册中心。

## 来源地址解析

系统接受完整 Git URL、SSH Git URL、GitHub 仓库路径和 GitHub 页面路径。

用户输入 `owner/repo` 或 `github.com/owner/repo` 时，系统解析为 GitHub 仓库克隆地址。

用户输入 GitHub tree 或 blob URL 时，系统解析仓库地址、分支或标签，以及仓库内子目录。

用户输入非 GitHub 的 HTTP Git URL 时，系统按普通 Git URL 处理，不解析 GitHub 子目录语义。

来源地址为空或格式不合法时，系统返回错误。

## 下载共享来源

桌面端允许用户输入 GitHub 来源并下载 Skill 项目到本地缓存。

系统把来源仓库克隆到应用数据目录下的来源缓存。缓存目录名称由仓库名称和来源摘要组成。

当缓存目录已存在时，系统执行快进拉取以更新本地来源。

来源地址指定分支或标签时，系统在克隆或拉取后检出该引用。

来源地址指定子目录时，系统把该子目录作为 Skill 项目根目录返回。

子目录不存在时，系统返回错误。桌面端把该下载项目标记为失败并保留错误信息。

## 发布计划

发布计划基于当前工作区快照生成，不直接写入远端仓库。

计划包含工作区根目录、仓库名称、可见性、技能来源目录下的文件清单、安装命令和发布检查项。

仓库名称来自工作区根目录名称。安装引用优先使用配置中的团队仓库地址，缺省时使用占位 GitHub 仓库引用。

私有发布计划提示用户确认仓库权限和评审负责人。

公开发布计划提示用户移除私有 URL、内部路径和公司内部流程描述。

发布计划包含已发现技能数量、运行审计、补充 README、打标签发布等检查项。

共享执行前，用户选择要共享的配置组。共享操作只上传该配置组包含的技能，并把该配置组写入共享仓库配置。

## GitHub 权限与交付方式

系统在自动共享前检测 GitHub 登录状态、目标仓库访问权限、默认分支、是否允许创建分支、是否允许创建 Pull Request，以及当前凭据是否支持 fork。

权限检测通过 GitHub CLI 或等价 API 能力完成。GitHub CLI 未登录、token scope 不足或 API 不可用时，系统返回权限待修复状态，并给出不会写入远端的恢复路径。

当用户对目标仓库具有写权限时，系统推荐创建共享分支并向默认分支提交 Pull Request。系统仍允许用户选择只创建本地分支，或在明确确认后直接推送共享分支。

当用户只有读权限且可以 fork 时，系统推荐 fork 仓库、推送到 fork 分支，并向上游仓库创建 Pull Request。

当用户没有目标仓库权限或 fork 不可用时，系统只生成本地分支、补丁说明、发布计划和手动命令，不执行远端写入。

自动 Pull Request 使用发布计划生成标题和正文。正文包含共享目标路径、配置组、技能数量、审计摘要、安装命令和发布检查项。

系统不代替 GitHub 的 review、合并、发布和访问控制。Pull Request 创建成功后，后续审核和合并在 GitHub 中完成。

## 共享目标路径

共享操作支持直接写入配置路径，也支持按项目名写入子目录。

直接写入模式使用来源地址中解析出的仓库子目录作为 Skill 项目根路径。

当直接写入模式的仓库子目录已经以当前 `sourceDir` 结尾时，系统把该子目录的父目录作为 Skill 项目根路径，避免生成重复的技能来源目录。例如来源地址为 `github.com/owner/repo/tree/main/definition/skills` 且 `sourceDir` 为 `skills` 时，系统写入 `definition/skills`，而不是 `definition/skills/skills`。

按项目名写入模式会把项目名追加到解析出的仓库子目录之后。

项目名为空且选择按项目名写入时，系统返回错误。

系统会清理目标路径中的空段、当前目录段和上级目录段，避免把项目写入非法相对路径。

## 共享执行

共享执行前，系统扫描当前工作区并解析共享目标。

共享执行前，系统根据权限检测结果确定交付方式。交付方式包括目标仓库分支 Pull Request、fork Pull Request、直接推送共享分支和本地分支。

共享执行使用隔离的共享工作树，不直接在用户当前工作区 checkout 中切换分支或提交。

当当前工作区不是同一个共享目标时，系统准备共享仓库检出目录。共享仓库检出目录位于应用数据目录下的共享工作树缓存，缓存目录名称由仓库名称和仓库地址摘要组成。

当检出目录不存在时，系统克隆目标仓库。检出目录存在时，系统拉取远端引用信息。

系统使用来源地址指定的分支；未指定分支时，系统优先使用远端默认分支，无法识别时使用 `main`。

当 GitHub tree URL 中的分支名包含斜杠时，系统根据远端分支列表匹配最长分支名，并把剩余部分作为子目录。

远端分支存在时，系统基于远端分支创建或重置本地分支，并执行快进拉取。

远端分支不存在时，系统创建本地分支，并在推送时创建远端分支。

同一共享仓库执行共享时，系统使用本地锁串行化写入，避免多个共享任务同时操作同一个本地 checkout。

## 文件同步

非同源共享时，系统在目标路径中创建目录，并确保目标路径下存在技能来源目录。

非同源共享时，系统按用户选择的配置组同步技能目录，并同步当前工作区共享资产目录。每个技能或共享资产在目标侧按相对技能来源目录的位置替换；目标技能来源目录中不属于本次配置组或当前工作区共享资产清单的其它目录保留。

同源共享时，系统不复制技能目录，只更新共享配置和 README，并提交当前工作区已有变更。

共享资产写入后带有 SkillOps 项目归属元数据。目标侧同名共享资产已存在且归属不是当前项目时，系统阻止共享，避免覆盖其它项目资产。目标侧旧共享资产没有归属元数据时，系统允许当前项目首次接管并写入归属元数据。

每个条目先复制到临时目录，复制成功后再替换目标目录；复制失败时保留原目标目录。

系统写入面向共享仓库的配置文件。该配置文件中的团队仓库地址指向可被安装的远端项目引用。目标已有配置文件时，系统保留其中其它配置组。当前项目的配置组以项目命名空间写入，例如 `project-a/default`，避免覆盖其它项目的 `default` 配置组。

目标路径没有 README 且当前工作区存在 README 时，系统复制该 README。目标路径已有 README 时，系统不覆盖整份 README。

系统向 README 写入或更新当前项目专属的 SkillOps 使用说明区块。该区块包含可见性、桌面端使用流程、配置组列表和 CLI 安装命令，不覆盖其它项目的 SkillOps 区块。

## 提交与推送

系统把目标路径变更加入 Git 暂存区。

当 Git 状态存在变更时，系统使用用户输入的提交信息创建提交。提交信息为空时，系统使用默认共享提交信息。

当 Git 状态没有变更时，系统不创建提交，并记录没有文件变更。

共享操作最后按交付方式推送共享分支到目标远端或 fork 远端。推送失败时，系统保留本地分支、提交哈希和手动恢复命令，并返回错误。

交付方式为 Pull Request 时，系统在推送成功后创建 Pull Request。Pull Request 创建失败时，共享结果保留本地分支、提交哈希、推送远端和可复制的手动命令。

交付方式为本地分支时，系统不推送远端。共享结果展示本地分支、提交哈希、目标路径和后续手动命令。

## 本地配置更新

共享前，系统把用户输入的远端仓库、共享目标模式和项目名保存到当前工作区配置。

本地配置中的团队仓库地址保留用户输入的远端来源。共享仓库中的配置使用可安装的远端项目引用。

## 成功标准

用户能够从 GitHub 来源下载一个 Skill 项目并作为本地项目打开。

用户能够生成私有或公开发布计划。

用户能够把当前 Skill 项目同步到指定 Git 仓库路径。

共享仓库中的 README 能说明桌面端使用流程、配置组和 CLI 安装命令。

用户能够在有写权限、只有读权限或没有权限的 GitHub 仓库场景中获得明确的推荐交付方式。

用户能够通过桌面端或 CLI 创建 Pull Request，或在权限不足时保留本地分支和手动共享材料。
